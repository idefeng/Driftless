## 1. 文档概述

- **项目名称**：Driftless
- **产品类型**：高精度跑步步频辅助工具（移动端 App）
- **核心定位**：专为追求极致节奏的跑者打造，基于底层硬件音频时钟调度，彻底消除传统软件在步频切换、循环时的微小延迟（Gap）与节拍漂移（Drift）。
- **设计原则**：Local-First（本地第一）。完全本地运行，零网络依赖，极致省电，保障绝对的低延迟与隐私安全。

## 1.5 命名释义：Driftless（供品牌 / Logo 设计参考）

> 本节用于统一团队对产品名内涵的理解，作为 Logo、视觉、文案的设计依据。

### 词义构成
**Driftless = Drift（漂移）+ -less（无 / 没有）= 「零漂移」**

- **Drift**：在本产品语境中，特指节拍器最致命的技术缺陷——*节拍漂移*：应用层定时器受系统调度、锁屏、CPU 负载影响，导致拍点随时间逐渐偏离，越跑越不准、产生累积误差。
- **-less**：后缀，表示「完全没有、彻底消除」。
- 合起来即产品的**核心技术承诺**：基于底层硬件时钟预调度，连续运行数小时**零累积漂移**（呼应 §3.1 的「累积漂移 = 0」指标）。名字本身就是卖点，无需额外解释即可向懂行的跑者传递「精准、稳定、不偏」。

### 双重寓意（设计可借力）
1. **技术层面**：No Drift —— 时钟不漂移，每一拍都钉在它该在的位置上。冷静、精密、可靠。
2. **跑者心流层面**：Drift 亦有「走神、涣散、随波逐流」之意；Driftless 可引申为**专注不涣散、心无旁骛、稳定在节奏里的心流状态**——跟着拍子跑，思绪不飘。这层含义温暖、有人格，适合用于情感化文案。

### 发音与书写规范
- 发音：/ˈdrɪftləs/（DRIFT-less，两个音节，重音在前）。
- 书写：首字母大写 **Driftless**，全小写 `driftless`（用于域名 / 包名 / handle）。**不拆写、不加空格、不写成 "Drift Less"**（避免与已存在的 "DriftLess" AI 建站站点视觉混淆）。

### 视觉联想关键词（给 Logo 设计的输入）

**整体基调：大众化、活泼、有活力，同时不失专业的可信感。**
目标用户不只是精英跑者，也包括大量想跑得更轻松、更有趣的普通跑步爱好者，因此视觉要**亲和、有动感、让人想点开**，而不是冷冰冰的极客工具。但「专业」不能丢——它是用户信任「这玩意儿真的准」的基础。两者的平衡点是：**用活泼的形式，承载精准的内核**。

- **核心意象**：一个精准对齐、却充满节奏跃动感的元素——例如一串等间距、有韵律起伏的节拍点 / 一条稳定却带弹性律动的脉冲波形 / 一个干净利落的步频刻度。形「准」，神「动」。
- **活力来源（体现大众化、活泼）**：
  - **明快、年轻的配色**——**主色锁定「阳光橙」**（明亮、温暖的暖橙，偏阳光感而非偏红的朱橙，传递能量、运动、热情，亲和又醒目），作为品牌的唯一主色贯穿 Logo、App 图标、关键按钮与强调元素；
    - *搭配建议*：主色橙 + 干净的中性色（白 / 浅灰底，深炭灰文字）即可，**不引入第二个高饱和强调色**（电光蓝、亮青等均不采用），保持视觉聚焦与品牌识别的统一；深色模式下用深炭灰 / 近黑背景衬托橙色，避免纯黑的工具感；
    - *主色色值（推荐）*：**阳光橙 `#FF8C2B`**（明亮暖橙）作为标准主色；配套深一档 `#F47216`（用于按下态 / 渐变深端、深色背景上的描边）、浅一档 `#FFB066`（用于浅色填充 / 高亮）。以上为推荐基准，设计阶段可微调，但确定后须全平台统一并写入设计规范（Design Token）。
  - **圆润的几何形**——圆点、圆角，比尖锐棱角更友好；
  - **节奏的跃动感**——通过元素的等距重复 + 大小 / 高低的轻微起伏，传达「跟着拍子跑起来」的愉悦，而非静止呆板。
- **专业来源（体现可信、精准）**：
  - **绝对的对齐与等距**——元素之间间隔精确相等，呼应「零漂移」；活泼归活泼，但底层结构必须工整；
  - **克制的造型**——简洁、可识别，能缩到 App 图标尺寸仍清晰；不堆砌细节。
- **可延展元素**：节拍点（圆点、竖线）、等间距网格 / 刻度、跑动的步幅律动、脉冲 / 声波纹。
- **反例（要避免传达的）**：
  - 别太冷 —— 避免纯黑灰、过度极客、像示波器 / 实验仪器的硬核工具感；
  - 别太花 —— 避免拟物化的卡通小人、繁复插画、糖果风的过度可爱，会削弱「精准可信」；
  - 别出现产品要消灭的东西 —— 漂移、倾斜、抖动、模糊、随机，不应进入视觉语言。
- **一句话给设计师**：像一款**好玩又靠谱的运动 App**——色彩明快、形态有节奏感，但每个元素都精确对齐，让人第一眼觉得有趣、再看一眼觉得专业。

### 命名注意事项（延续自商标初筛）
- 「Driftless」亦是美国中西部一处地理名词（"Driftless Area" 无冰碛区），故存在披萨店等无关同名；好处是健身 / 音频品类内无撞名，代价是**作为地理通用词，商标显著性偏弱**。
- **建议**：正式注册前，搭配一个独特的图形 Logo + 可能的后缀（如 "Driftless — Run Cadence"）来增强显著性与可注册性，并由商标律师就软件类目做 knockout search。

## 2. 核心用户痛点

- **痛点 1**：普通软件播放长音频循环时，由于解码、缓冲（Buffer Latency）或音频首尾自带静音，导致循环衔接处出现几十毫秒的卡顿，打乱跑步节奏。
- **痛点 2**：应用层定时器（如 `setInterval`）极易受系统后台调度影响，锁屏或 CPU 负载高时节拍点发生漂移。
- **痛点 3**：跑步途中大汗淋漓或手机在臂包内，传统的滑动条（Slider）或小按钮极难进行精准的步频微调。

## 3. 功能需求 (Functional Requirements)

### 3.1 高精度音频合成引擎（核心功能）

**技术机制**
禁止使用长音频循环方案。必须采用**短采样预调度机制（Look-ahead Audio Scheduling）**：应用层维护一个「调度窗口」（建议 100–200ms），在每个调度周期内，将未来窗口内所有拍点按硬件音频时钟（如 `AudioContext.currentTime` / `AVAudioEngine` 采样时钟 / Android `AAudio` 时间戳）的**绝对时间戳**预先排入播放队列。所有拍点间隔基于采样率精确计算，**绝不使用累加定时器**，从根本上杜绝累积漂移。

**节拍范围**
支持 **100 BPM 至 250 BPM**（每分钟步数）的精准调节，步进 1 BPM。

**精准度指标（已拆分为两个独立维度，不可混为一谈）**

| 指标 | 定义 | 目标值 | 备注 |
|---|---|---|---|
| 拍间间隔抖动（Jitter） | 相邻两拍实际间隔与理论间隔之差 | **< 1 ms** | 锁屏、后台、系统高负载下均需满足 |
| 累积漂移（Drift） | 连续运行后总时长误差 | **0（采样级对齐，无累积）** | 连续运行 ≥ 2 小时验证 |
| 绝对输出延迟（Latency） | 从调度到声音真正出声的固定时延 | **硬件相关，约 5–30 ms** | *为系统音频缓冲固有延迟，无法消除；对节拍器为恒定偏移，不影响节奏一致性* |

> ⚠️ **说明**：原始需求中的「±0.1 ms」实为对「绝对延迟」的误用。声音从调度到出喇叭必经系统音频缓冲，物理上无法达到 0.1ms；且 0.1ms 远低于人耳节奏可分辨阈值（约 1ms 量级）。本产品真正需要保证的是**间隔抖动 < 1ms 且零漂移**，绝对延迟作为硬件固有项单独列出。

**音效选择**
内置不少于 **3 种**高穿透力、低延时的节拍音效（如：电子嘀声、木鱼声、标准节拍器鼓点）。音效文件须满足：
- 首尾**零静音段**（精确剪裁到波形起止点）；
- 单个采样时长尽量短（建议 < 80ms），降低解码与调度开销；
- 提供归一化音量，避免不同音效响度差异。

### 3.2 音频共存：与音乐 / 视频同时播放（Mix-With-Others）（核心功能）

**目标**
节拍音**不得独占音频设备**。用户在听音乐、听播客、看 YouTube / B 站视频的同时，节拍声叠加播放于其上，跑者既能跟节奏跑，又不至于无聊。这是 Local-First 之外的第二条硬性体验底线。

**默认行为**
默认进入「共存模式」：本 App 仅向音频输出**叠加（mix）**自己的节拍采样，**绝不打断、绝不暂停**其它 App 的音频。设置中可切换为「独占模式」（不与他人混音，适合纯节拍训练）。

**平台实现要点**

- **iOS（AVAudioSession）**
  - 类别使用 `.playback`（保证后台与静音键开关下仍能出声），**并附加 `.mixWithOthers` 选项**，即可在保留后台播放能力的同时与其它 App 共存。
  - 选用 `.mixWithOthers` 后，本 App **主动放弃** Now Playing / 远程控制权（交还给音乐 / 视频 App）——这正是我们想要的，但它会直接影响 §3.3 的锁屏媒体栏方案（见下）。
  - 可选「轻压（Ducking）」：附加 `.duckOthers` 会在整个会话期间压低他人音量。注意对**持续运行**的节拍器而言，音乐会被全程压低，多数用户不希望如此，故 **`.duckOthers` 默认关闭**，仅作为设置项提供。

- **Android（AudioFocus / AudioAttributes）**
  - **关键：不申请独占音频焦点（`AUDIOFOCUS_GAIN`）**，否则会暂停用户正在播放的音视频。直接通过 `AudioTrack` / `SoundPool` 叠加输出即可与系统其它音频自动混音。
  - 若产品决定提供「轻压」选项，则改为申请 `AUDIOFOCUS_GAIN_TRANSIENT_MAY_DUCK`；默认不申请、纯叠加。
  - `AudioAttributes` 的 usage 建议设为 `USAGE_ASSISTANCE_SONIFICATION`（提示音性质），content type 设为 `SONIFICATION`，以获得合适的混音与不被随意压低的表现。

**与「音效选择」(§3.1) 的联动**
混音后节拍声需在音乐 / 人声背景中**清晰可辨**。因此 §3.1 的音效不仅要「低延时」，更要「高穿透 / 高频段区分度」（如电子嘀声、木鱼的瞬态尖峰），并建议提供**独立于系统媒体音量的节拍音量条**，让用户单独调高节拍相对音乐的响度。

**验收标准**
- 打开 Apple Music / Spotify / YouTube 播放，再启动本 App：音乐 / 视频**继续不间断播放**，节拍声叠加其上；
- 切换共存 ↔ 独占模式，行为符合预期；
- 来电等真中断结束后，节拍恢复，且不影响其它 App 的音频续播。

### 3.3 跑者友好型交互界面 (UI/UX)

**大面积微调盲操**
- 主界面提供占据屏幕**各 1/4 面积**的超大 `+1` 与 `-1` 按钮，方便跑步中盲操微调步频。
- 长按按钮支持连续步进（加速递增），便于大幅调整。

**实体音量键联动（分平台标注）**
在 App 前台运行时，可通过手机物理音量键直接进行步频的 ±1 或 ±5 调节（可在设置中切换「音量 / 步频」映射模式）。

> ⚠️ **平台差异（务必在排期前确认）**：
> - **Android**：完整支持，可拦截音量键事件并屏蔽系统音量 UI。
> - **iOS**：仅能通过监听 `outputVolume` 变化**间接感知**音量键被按下，无法干净地屏蔽系统音量 HUD，且将音量键重映射为他用历来易触发 App Store 审核拒绝。**此功能在 iOS 上标记为「待审核确认」，立项前需复核当前《App Store Review Guidelines》**。可考虑的兜底方案：仅在耳机线控 / 蓝牙媒体键上做映射，或在 iOS 放弃此功能、改用屏幕大按钮。

**常亮 / 锁屏显示**
- 主界面大字号步频数字（BPM），高对比度配色（适配夜跑与强光下跑步）。
- 屏幕常亮（Keep Awake）可在设置中开关，默认开启。
- 集成系统**锁屏实时活动 / 通知**显示当前步频，锁屏状态下亦可直观查看。

> ⚠️ **与 §3.2 共存模式的冲突（重要）**：原方案设想「集成到系统媒体控制栏并在其中 ±1 调节」。但在共存模式下，本 App 已主动放弃 Now Playing / 远程控制权（归正在播放的音乐 / 视频 App 所有），**系统媒体栏同一时刻只能由一个 App 拥有**，因此节拍器**无法**再占用媒体栏。
> - **iOS**：改用 **Live Activities + 灵动岛**展示步频并提供 ±1 按钮（Live Activity 独立于 Now Playing，不冲突）。
> - **Android**：改用**本 App 自有的常驻通知**（前台服务通知）承载步频显示与 ±1 操作，而非抢占系统媒体会话。
> - 仅当用户切到「独占模式」时，才可考虑由本 App 接管媒体栏。

### 3.4 进阶功能：结构化训练流（训练模式）

**自定义间歇步频计划**
用户可自由组合训练流水线，每段定义「时长 + 目标 BPM」。
> 示例：5 分钟热身（170 BPM）→ 20 分钟巡航（180 BPM）→ 5 分钟冲刺（190 BPM）。

**无缝阶梯过渡（语义需明确定义）**
从一段切换到下一段时，音频时钟必须无缝衔接，绝不允许出现物理撕裂感或多余的空拍。本产品采用以下明确语义（**默认且推荐**）：

> **「自然收尾，立即换算」**：当前段的最后一拍按当前 BPM 完整走完；**该拍发声的时刻即作为下一段的起拍基准**，此后所有拍点立即按新 BPM 的间隔重新排布。段落边界（如「第 5 分钟」）允许有不超过「一拍间隔」的浮动，以换取绝对无缝的听感。

> ❗ 与之相对的「严格对齐时间边界」方案（强制在第 5 分钟整切换，可能截断当前拍）会引入听感上的撕裂，**本产品明确不采用**。该选择已在此写死，避免开发期反复。

可选增强（标注为 P1，非首版必须）：**渐进过渡（Ramp）**，在切换前数秒内将 BPM 从 170 平滑线性过渡到 180，更贴近真实加速体感。

## 4. 非功能需求 (Non-Functional Requirements)

### 4.1 性能与功耗

**后台存活率**
- **iOS**：配置 `Audio` 后台模式（AudioPlayback），保持音频会话活跃以最大化后台存活。**注意：`.playback + .mixWithOthers`（见 §3.2）在保留后台播放能力的同时实现共存，二者不冲突**。
- **Android**：使用前台服务（Foreground Service，类型 `mediaPlayback`）+ 常驻通知，并引导用户在系统电池优化中将本 App 加入白名单。**前台服务负责「保活」，与「是否申请音频焦点」是两件独立的事**——共存模式下保活照常，但不申请独占焦点（见 §3.2）。
- 目标：被系统杀后台的概率降到最低。

**极低能耗（指标按来源拆分，避免相互矛盾）**

| 耗电来源 | 目标 | 说明 |
|---|---|---|
| 音频引擎（纯计算 + 短采样调度） | **单小时 < 1%** | 不解码大文件，CPU 占用极低 |
| 屏幕常亮 | 独立计量，不计入引擎指标 | 常亮屏是耗电大头，由用户开关决定 |
| 锁屏实时活动 / 通知刷新 | 独立计量 | 控制刷新频率以降低开销 |

> ⚠️ **说明**：原指标「单次跑步 1 小时 < 3%」在「屏幕常亮 + Live Activities」开启时无法成立——耗电大头是屏幕而非音频。故将「引擎计算」与「显示部分」分别约束。**「关闭常亮、锁屏运行」场景下，整机 1 小时耗电目标 < 3%**；常亮场景另行计量。

**音频中断处理（按共存 / 独占模式区分）**
必须定义以下场景的行为：
- **其它 App 开始播放音乐 / 视频**：在**共存模式下绝不暂停**，节拍声继续叠加播放（这正是 §3.2 的目的）；仅在「独占模式」下才考虑暂停或被暂停。
- **来电 / 闹钟等真中断**：节拍器暂停（iOS 下即便是 `.mixWithOthers` 会话，来电仍会触发系统中断）；中断结束后，根据设置「自动恢复」或「等待用户手动恢复」。训练模式下需正确记录并续接剩余时长，且恢复时不影响其它 App 的音频续播。
- **耳机插拔 / 蓝牙断连**：默认暂停（避免突然外放），重连后提示恢复。
- **音频路由切换**：切换扬声器 ↔ 耳机时不得产生爆音或丢拍。

**蓝牙延迟说明（原文档缺失，新增）**
蓝牙耳机普遍存在 **100–300ms** 固定输出延迟。对纯音频节拍器而言，这是**恒定偏移，不影响节奏一致性**，可接受。但：
> ❗ 若未来引入**触觉反馈（震动）**辅助，音频（经蓝牙延迟）与震动（本机即时）将明显错位。届时需对蓝牙音频路径做延迟补偿，或在蓝牙连接时自动关闭「音震同步」。本版规格预先声明此约束。

### 4.2 数据与隐私 (Local-First)

**零网络依赖**
应用启动、运行、配置保存均在本地完成，无任何网络请求，确保越野跑等弱信号 / 无信号环境下绝对稳定。无账号体系、无遥测、无广告 SDK。

**数据存储**
用户的训练计划与常用步频参数通过本地轻量级存储（如 SQLite 或 MMKV）持久化。需支持训练计划的本地导入 / 导出（如 JSON 文件），便于跑者间分享与备份，同时不破坏「零网络」原则。
